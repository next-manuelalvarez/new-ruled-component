1. Crear una rama en un fork del monorepo

  • Crear un fork del repositorio bbva-web-components-app
  • Crear un rama en el fork git branch -b feature/jira-cod[opcional] + description
  • Una vez concluido el desarrollo lanzar una pull request de esta rama a a master del repositorio original

2. Crear un nuevo componente - Cells CLI

Cells CLI 3.0.0 es la versión mínima requerida

cells lit-component:create [name] [name-space]

> cells lit:component bbva-button-basic @bbva-web-component

(El namespace de los componentes básicos es @bbva-web-components)

3. Dependencias - package json

  • Los campos description y keywords son obligatorios
    • Description - (string) descripción del componente
    • Keywords - (array) colección de palabras clave descriptivas del componente en custión
  • El campo scripts debe llevar los siguiente comandos :
"scripts": {
	"test": "npx karma start ../../_karma/conf.js",
	"test:compatibility": "npm run test -- --force-polyfills --compatibility all",
	"test:compatibility:watch": "npm run test:compatibility -- --auto-watch=true --single-run=false",
	"test:prune-snapshots": "npm run test -- --prune-snapshots",
	"test:update-snapshots": "npm run test -- --update-snapshots",
	"test:watch": "npm run test -- --auto-watch=true --single-run=false"
}

Los comandos test y test-watch lanzan las baterías de test unitarios. Con el primero se obtiene un resultado de los test y el porcentaje de cobertura de los mismo. El segundo sirve para mantener los test lanzados mientras se desarrollan nuevas pruebas.

Se deben realizar test de snapshot mediante los comandos anteriormente descritos. Cada vez que se haga un cambio en el componente es recomendable actualizar estos snaphots. (más información en el siguiente apartado 4.Snapshots)

4. Snapshots

Mediante npm o yarn deben realizarse los test snapshot, para ello son necesarios los siguiente pasos:

  1. Añadir los siguientes test en el fichero de test unitarios :
  1. Ejecutar el script del package.json test
  1. En el caso de que el componente desarrollado sea un widget añadir la configuración de karma.js

suite('Semantic Dom', () *=>* {
	test('DOM - Structure test', () *=>* {
		assert.dom.equalSnapshot(el);
	});
	
	test('SHADOW DOM - Structure test', () *=>* {
		assert.shadowDom.equalSnapshot(el);
	});
	
	test('LIGHT DOM - Structure test', () *=>* {
		assert.lightDom.equalSnapshot(el);
	});
});

5. Styles

Todas las propiedades susceptibles de ser editadas desde otro componente deben obtener su valor de una variable. A su vez esta variable debe llevar un fallBack que obtendrá de foundations.

Todas las medidas deben ser relativas, se pueden utilizar la función rem para convertir pixeles a rems o la función caclSpacer, que puede recibir varios parámetros.

margin: rem(16px)
@function calcSpacer($spacerValue, $unit: rem, $base-font-size: 16, $baseGridSpacerValue: 8)
margin: calcSpacer(24px, rem)

6. Enviar Pull Request

Todos los componentes de este repositorio bbva-web-components-app pueden ser editados siempre y cuando se cumplan dos premisas :

  • Los cambios realizados están validados y publicados por federación de diseño en BBVA Experience
  • Todos los cambios deben ser enviados mediante pull reques siguiendo el paradigma fork - branch

Las reglas básicas para la aprobación de pull request se pueden consutar en Team Spaces del Gobierno de Componentes, siendo las más importantes las sigueintes :

  • Todas las modificaciones visuales deben estar publicadas previamente en BEX
  • La cobertura de test unitarios debe ser de al menos el 80% y nunca inferior a la existente antes de la aportación
  • La pull request debe incluir un solo commit (commit squash)
  • Los mensajes de commit deben seguir las reglas de convntional commits
